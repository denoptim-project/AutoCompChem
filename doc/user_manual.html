Introduction

This tool kit for the Automation of Computational Chemistry (AutoCompChem) is developed with the aim of increasing efficiency, effectiveness, and reproducibility of computational tasks such as modeling and evaluation of chemical species.
In practice, AutoCompChem is the interface to the collection of methods and objects stored in the UiBkvant library: a Java library meant to host methods and objects developed day by day to automatize tasks such as creation of input files, simple operations on molecular systems, analysis of molecular geometries, and so on.
In the present document you can find information on how to use the various tools made available by UiBkvant. Please refer to UiBkvant documentation for more details on UiBkvant classes.

Building and Testing AutoCompChem

To compile AutoCompChem you just have to run the building script: 

./build_test.sh
and follow the instructions. Java 1.7.0_07 or higher is required.
After the script has compiled AutoCompChem and link the libraries of CDK, guava and UiBKvant into a single jar file, a list of test will be executed. Each test is provided with input files and specific a script that evaluates the outcome and returns a passed/not-passed grade.

Running AutoCompChem

AutoCompChem accepts one, and only one, argument that is the name of the parameters file defining "what" and "how". Thus, this is how you run: 

java -jar $yourpath/AutoCompChem.jar [parameters_file]
In general, should AutoCompChem encounter an error of any kind error messages will be printed both on stderr and stdout. Such error messages are meant to be as self-explanatory as possible. If you really cannot understand what's going on, be kind and contact the author.

The Parameters File

The parameters file is a formatted text file where all the information needed to perform a task is reported. All keywords, data, file names, links and anything else needed to perform a task is stored into the parameters file as organized text. The content of a parameters file can be divided into these main blocks (see ParameterStorage):

  • Keywords: a single line starting with a keyword (no leading spaces) followed by a separator(":"), and continuing with one or more arguments.
  • Labeled Blocks: one or more lines identified by a defined beginning (label $START), a keyword (a single word immediately following the beginning label), and an well defined end (label $END). Labeled blocks are used to provide complex information to AutoCompChem, that is, information that is not interpreted by AutoCompChem in the first place, but is sent and deployed by one of the tools/routines made available in subsidiary libraries (i.e., uibkvant). The special meaning of the dollar sign can be escaped like in a bash script by using the backslash sign (\) in front of any special character. Foe example, only the second dollar sign of the string "\$end$end" is interpreted as such, while the first is escaped. As a result, AutoCompChem will interpret the second instance of "$end" as the end of a labeled block, while the first "$end" is threated as bare text.
  • Comments: each line starting with "#" is ignored unless is part of a labeled block.

The actual list of keywords and labeled block that are required by each task, depends on the single task. Therefore, after having had a look at the list of available tasks, you'll find more information looking at the documentation of the single task that you wand to perform with AutoCompChem.

Examples of parameter files can be found in the test folder for most of the task. Please remember that the tests are meant to trigger potential problems and might not always be sensible from the scientific point of view. In fact, some tests are even meant to make AutoCompChem return error messages. So, run the tests and check their outcome before doing copy-paste of parameters file.

Tasks for AutoCompChem

The following list of tasks are currently implemented in AutoCompChem. Note that, at the moment, only one of these tasks can be executed per each run.

  1. Molecular Modeling
    1. Software-independent:
      1. evaluate the output of a generic computational task against a list of known errors (keyword EvaluateGenericOutput)
    2. Gaussian
      1. Prepare input (keyword PrepareInputGaussian). Also with generation of customized, atom-specific basis set (see also GenerateBasisSet).
      2. Evaluate output (keyword EvaluateGaussianOutput)
      3. Troubleshoot Gaussian calculation (keyword FixAndRestartGaussian)
    3. NWChem
      1. Prepare input (keyword PrepareInputNWChem). Also with generation of customized, atom-specific basis set (see also GenerateBasisSet).
      2. Evaluate output (keyword EvaluateNWChemOUTPUT)
      3. Troubleshoot NWChem calculation (keyword FixAndRestartNWChem)
      4. Extract Cartesian coordinates from NWChem output file (keyword ExtractTrajectoryFromNWChemOutput)
      5. Extract only the last geometry from NWChem output file (keywords ExtractLastGeometryFromNWChemOutput.
    4. QMMM
      1. Prepare input (keyword PrepareInputQMMM)
    5. Spartan
      1. Prepare input (keyword PrepareInputSpartan)
      2. Extract all output geometries (keyword ExtractGeometriesFromSpartanTree)
      3. Extract last output geometry (keyword ExtractLastGeometryFromSpartanTree)
    6. VibModule
      1. extract force field parameters from VibModule's output file (keyword ExtractVibModuleForceConstants)
  2. Handling Molecules
    1. Modification: systematic alteration of molecules.
      1. Change geometry of all or only of some specific atoms (keyword ModifyGeometry).
      2. Remove dummy atoms from a molecular representation (keyword RemoveDummyAtoms).
      3. Remove molecular fragments from a molecular representation (keyword PruneMolecules).
      4. Adjust connectivity (keyword RicalculateConnectivity).
      5. Alter bonding of specific elements (keyword AddBondsForSingleElement).
      6. Mutate the elemental symbol of some atoms (keyword MutateAtoms).
      7. Reorder list of atoms (keyword ReorderAtomList).
    2. Comparison: for handling a pair of molecules and evaluate similarities and differences.
      1. Compare two molecules and atom-atom mapping (keyword CompareTwoMolecules)
      2. Compare only the geometry of two molecules (keyword CompareTwoGeometries)
      3. Compare only the connectivity of two molecules (keyword CompareTwoConnectivities)
      4. Sorts molecules according to SDF field (keyword SortSDFMolecules)
    3. Analysis: determination of descriptors
      1. Measure inter atomic distances, angles and dihedrals (keyword MeasureGeomDescriptors)
      2. Find atom clashes (keyword AnalyzeVDWClashes)
      3. Analyze chelation (keyword AnalysisChelates)
More details on each task, and the list of parameters needed to perform it, are provided in the following sections.

PrepareInputGaussian

This task allows you to prepare input files automatically. A so-called JOBDETAILS file has to be provided with the details defining all the steps of the job. The format of such file is described in the documentation of the GaussianJob class. The parameters needed to perform this task are:

  • INFILE: name of the structure file (i.e. path/name.sdf).
  • JOBDETAILS: formatted text file defining all the details of a GaussianJob. Usually the jobdetails txt file is the file used to generate the input file (name.inp) for Gaussian. The definition of the format of jobdetails files can be found in GaussianJob documentation. Examples of jobdetails files are also available in the test folder. In alternative, use keyword HEADER with a labeled block of lines (i.e., a bunch of text starting with the $START label and finishing with the $END label). The text in the labeled block is used as header of the Gaussian input file. In this header only Gaussian's 'Link0' and 'Route' sections should be included. For comments, charge and spin multiplicity, see below.

    WARNING! This header is treated as a pure string and no check of its format will be performed. Thus if the header has wrong format, so will have the output file. Moreover, by using a fixed header no modification of such header can be performed by other tools of available to AutoCompChem (i.e. GaussianReStarter). For instance, name of checkpoint file, memory, nodes, processors, and everything that is specified in the header CANNOT BE EDITED!

  • (optional) VERBOSITY verbosity level.
  • (optional) OUTNAME name of the output file (.inp for Gaussian)
  • (optional) COMMENT comment line for the output file
  • (optional) CHARGE the charge of the chemical system. This parameter is mandatory only when the charge is not defined in the SDF input file.
  • (optional) SPIN_MULTIPLICITY the spin multiplicity of the chemical system. This parameter is mandatory only when the multiplicity is not defined in the SDF input file.

FixAndRestartGaussian

AutoChompChem can be used to automatize the evaluation of the outcome of a Gaussian job and, in case of an error, generate the input for a new job with settings dependent on the results of the first job. In practice, AutoChompChem can be used to automatize standard operations aimed to govern Gaussian jobs, and to generate a dynamic sequence of jobs. In the following list of parameters are required with using the FixAndRestartGaussian keyword (see GaussianReStarter for further information including the type of error fixing action that can be performed).

  • INFILE: path or name of the output from Gaussian (i.e. name.out).
  • NEWINPFILE: path or name of the new input file (i.e. name.inp) to be generated for Gaussian. The input will consider that a checkpoint file with the very same root name (i.e., name.chk) is to be used. This means that a copy of the old checkpoint file has to be made and placed properly before the new inp is submitted to Gaussian.
  • JOBDETAILS: formatted text file defining all the details of the Gaussian job originally submitted to Gaussian.
  • GO3ERRORS path to the folder storing Gaussian known errors.
  • VERBOSITY verbosity level.

PrepareInputNWChem

This task allows you to automatically prepare input files for NWChem. A so-called JOBDETAILS file has to be provided with the details defining all the steps of the NWChem job. The format of such file is described in the documentation of the NWChemJob (see also NWChemInputWriter). The parameters required to perform this task are:

  • INFILE: name of the structure file (i.e. path/name.sdf).
  • JOBDETAILS: formatted text file defining all the details of a job. The definition of the format of jobdetails files can be found in NWChemJob's documentation. Examples of jobdetails files are also available in the test folder.
  • (optional) MULTIGEOMNWCHEM: use this parameter to provide names for the geometry/ies (all space-free names listed in one single line, space-separated). When this parameter is used all the geometries in the input molecular structure file (i.e., SDF or XYZ) are written into the first task of the NWChem job with unique atom tags. Basis set and constrains are also generated for all the geometries. This parameter can also be specified within a job details file (see below) as a task-specific parameter in the first task (see also NWChemTask).
  • (optional) USESAMETAGSFORMULTIGEOM: imposes to use the same atom tag through all geometries. We assume that all the geometries are consistent (i.e., same atom list). This parameter can also be specified within a job details file (see below) as a task-specific parameter in the first task (see also NWChemTask).
  • (optional) VERBOSITY verbosity level.
  • (optional) OUTFILE name of the output file (i.e., the input for NWChem). If this option is omitted the default name of the output is build from the root of the structure file name.
  • (optional) TITLE title line for the output file
  • (optional) CHARGE the charge of the chemical system. This parameter is mandatory only when the charge is not defined in the SDF input file.
  • (optional) SPIN_MULTIPLICITY the spin multiplicity of the chemical system. This parameter is mandatory only when the multiplicity is not defined in the SDF input file.
  • (optional) COORDINATESTYPE the type of coordinates (Cartesian or internal) to be used for the input file of NWChem.
  • (optional) FREEZEIC define frozen internal coordinates (i.e., constants) To identify the IC to freeze, SMARTS queries are used. A multi line block (see ParameterStorage) can be used defining one SMARTS query per line.
  • (optional) FREEZEATM define frozen atoms when using Cartesian coordinates. SMARTS queries are used to identify atoms to freeze; one SMARTS per line.
  • (optional) ZCOORD can also be used to freeze redundant internal coordinates and eventually release such constraints. This only works if the geometry is written with Cartesian coordinates and NWChem uses the "autoz" module to generate the redundant internal coordinated (NB: the "autoz" keyword is active by default) SMARTS queries are used to identify the specific classes of internal coordinates (IC) that will be listed in the ZCOORD directive. Single-atom SMARTS (i.e., must be enclosed in square brackets, eg. [C]) are used to define a single atom type. Combinations of 2, 3, or 4 whitespace-separated single-atom SMARTS are used to refer to stretching, bending, or torsion types of ICs. All n-tuple (n=2,3,4) of matching atoms that are also properly connected are considered. After the 2/3/4 single-atom SMARTS the user can add one numerical value and/or the constant keyword that will apply to all ICs matched by the combination of single-atom SMARTS. A multi line block (see {@link uibkvant.parameters.Parameter}) is used to define one set of SMARTS queries (plus additional keywords- related to that class of IC) per each line. 
    $STARTZCOORD:
[C] [$(C(~[N])~C)] 1.5 constant
[*] [$(C(~[N])~C)] [*] 120.000 constant
[*] [$(C(~[N])~C)] [C] [*]
$END

Note that specifying these options in the parameters file fed to AutoCompChem will make these options take effect from the first NWChemTask of the generated NWChemJob. Also, most of the options (see NWChem user manual for further details) will be effective until new instructions (i.e., keyword, directive) are given into the NWChemJob.

To specify task-specific parameters, i.e., parameters that are meant for a specific NWChemTask in the NWChemJob, use the jobDetails file (see NWChemTask).

EvaluateNWChemOUTPUT

This task allows to read and analyze the output of an NWChem job evaluating whether the job has terminated normally or not. In the latter case, the error messages from NWChem are compared with a list of known errors to identify the problem (see NWChemOutputHandler). The evaluation requires the following parameters:

  • INFILE: path or name of the output from NWChem i.e. name.out).
  • NWCHEMERRORS: path to the folder storing NWChem known ErrorMessages.
  • VERBOSITY: verbosity level.
  • (optional) JOBDETAILS: formatted text file defining all he details of a NWChemJob. Usually the jobdetails txt file is the file used to generate the original input file (name.nw) for the NWChem job that produced the output file under analysis (see NWChemInputWriter).
  • (optional) OUTFILE: name of the file where output molecular structure will be saved. If multiple output file are produced the string given via this option will be used a root of all output file names
  • (optional) OUTFORMAT: (default XYZ): format for molecular structure file. Currently available options: XYZ, SDF, both XYZ and SDF.
  • (optional) PRINTLASTGEOMETRY: imposes to print the last geometry after evaluation of an output file.
  • (optional) ANALYSE: types of analysis to perform. Acceptable values are:
    • ENERGY: extracts the energy and, if available, thermochemical corrections;
    • IMGFREQ: checks for imaginary frequencies (up to 5) and identifyes the state as minimum, transition state, or higher order saddle point.
  • (optional) LOWESTFREQ: overwrites the default lowest (cm-1) frequencies accepted: values below the one given will be considered zero.

FixAndRestartNWChem

This task allows to read the output of an NWChem job, evaluate if the calculation terminated with an error, and, if the error is recognized as a known error, generate an NWChem input file corresponding to a defined "error-fixing action" aiming to fix the problem and achieve the completion of the NWChem job (see NWChemReStarter). The execution of the this task requires the following parameters:

  • INFILE: path or name of the output from NWChem (i.e. name.out).
  • NEWNWFILE: path or name of the new input file (i.e, name.nw) to be generated for NWChem. The input will consider that a database file with the very same root name (i.e., name.db) is to be used. This means that a copy of the old database file has to be made and placed properly before the new name.nw is submitted to NWChem. In alternative, the option NWCDATABASE can be used to specify a different name for the database file.
  • JOBDETAILS: formatted text file defining all the details of the NWChem job originally submitted to NWChem. The format of a jobdatails file is described in NWChemJob's documentation.
  • NWCHEMERRORS path to the folder storing NWChem known errors. The format of the files defining errors is described in ErrorMessage's documentation. Acceptable error-fixing actions are described here.
  • (optional) OUTFORMAT: (default XYZ): format for molecular structure file. Currently available options: XYZ, SDF, both XYZ and SDF.
  • (optional) ANALYSE: types of analysis to perform. Acceptable values are:
    • ENERGY: extracts the energy and, if available, thermochemical corrections;
    • IMGFREQ: checks for imaginary frequencies (up to 5) and identifyes the state as minimum, transition state, or higher order saddle point.
  • (optional) LOWESTFREQ: overwrites the default value (cm-1) acceptable for frequencies: values below the one given will be considered zero.
  • (optional) NWCDATABASE name of NWChem's database file that holds runtime settings and data of an NWChem job (i.e., the checkpoint file in NWChem)
  • VERBOSITY verbosity level.

PrepareInputQMMM

This task allows to generate the input files (i.e., both the ".dat" with the settings and the ".crd" with the geometry) for running a calculation with the QMMM software developed by Hai Lin and Donald G. Truhlar (see also QMMMInputWriter)). To prepare the input files the following parameters are used:

  • INFILE: name of the structure file (i.e. path/name.sdf).
  • JOBDETAILS: formatted text file defining all the details of a QMMM job. The definition of the format of jobdetails files can be found in QMMMJob's documentation.
  • (optional) VERBOSITY verbosity level.
  • (optional) OUTFILE name of the output file (i.e., input for QMMM). If this option is omitted the default name of the output is build from the root of the structure file name.
  • (optional) TITLE title line for the output file
  • (optional) CHARGE the charge of the chemical system. This parameter is mandatory only when the charge is not defined in the SDF input file.
  • (optional) SPIN_MULTIPLICITY the spin multiplicity of the chemical system. This parameter is mandatory only when the multiplicity is not defined in the SDF input file.
  • (optional) QM_SOFTWARE defined the software that will perform quantum molecular mechanics calculations (with version ID). Default is GAUSSIAN-09.
  • (optional) MM_SOFTWARE defined the software that will perform classical molecular mechanics calculations (with version ID. Default is TINKER-6.3.3.
  • (optional) QMATOMS_ID define atoms belonging to QM region by listing atom ID (1-based atom number in the input molecular representation)
  • (optional) QMATOMS_SMARTS define atoms belonging to QM region using SMARTS queries. A multi line block (see ParameterStorage) can be used defining one SMARTS query per line.
  • (optional) CAPATOMS_SMARTS define cap atoms using SMARTS queries and cap atom specification. A multi line block (see ParameterStorage) can be used defining each line one SMARTS and the cap atom properties (name, atom type, scale factor) used to define the cap atom that replaces the atom matched by the SMARTS query. The syntax for each lice is as follows: [SMARTSquery] [cap atom specs]
  • (optional) FREEZEATM define frozen atoms. SMARTS queries are used to identify atoms to freeze. A multi line block (see ParameterStorage) can be used defining one SMARTS query per line.
  • (optional) ATOMTYPESFILE provides a file from which atom types are read and specifies the format for reading such file. The syntax for this option is as follows: [filename] [format]

PrepareInputSpartan

This task allows to generate the work environment for running Spartan (tested with Spartan14 under Linux). To perform this task use the following parameters (see also SpartanInputWriter):

  • INFILE: name of the structure file (i.e. path/name.sdf).
  • KEYWORDS: the list of keywords for Spartan
  • (optional) CHARGE the charge of the chemical system. This parameter is mandatory only when the charge is not defined in the SDF input file.
  • (optional) SPIN_MULTIPLICITY the spin multiplicity of the chemical system. This parameter is mandatory only when the multiplicity is not defined in the SDF input file.
  • (optional) VERBOSITY verbosity level.
  • (optional) OUTFILE name of the output file (input for Spartan). If this option is omitted the default name of the output is build from the root of the structure file name given in the INFILE parameter.
  • (optional) ROTATABLEBONDS the definition of the rotatable bonds. can be a pathname or a single/multi line block (see ParameterStorage) in which each line contains a bond-matching SMARTS query and additional options/keywords (for instance, the fold number to use for the bonds matched by the SMARTS query). For Example: 
    $STARTROTATABLEBONDS:
[C]~[#6] 3
[#7]!@~[#6] 3
[$([#6]1~[#7]~[*]~[*]~[*]~[*]1)] 2
$END
  • (optional) CONSTRAINTS definition of the constraints. Single-atom SMARTS (i.e., must be enclosed in square brackets, eg. [C]) are used to define a single atom type. Combinations of 2, 3, or 4 whitespace-separated single-atom SMARTS are used to refer to stretching, bending, or torsion types of ICs. All n-tuple (n=2,3,4) of matching atoms that are also properly connected are considered. After the 2/3/4 single-atom SMARTS the user can add one numerical value and/or text-type options that will apply to all ICs matched by the combination of single-atom SMARTS. A multi line block (see ParameterStorage) is used to define one set of SMARTS queries (plus additional numerical or text) per each line. For example: 
    $STARTZCOORD:
[C] [$(C(~[N])~C)] 1.512
[*] [$(C(~[N])~C)] [*] 120.000 optionA option B
[*] [$(C(~[N])~C)] [C] [*]
$END
  • (optional) ATOMSPECIFICKEYWORDS definition of atom specific keywords. SMARTS can be used to identify specific atoms and add the atom specific keyword to the input file. One SMARTS-based rule per line. Multi line block can be used. To specify the keyword use the following syntax: 
    <SMARTS> <part1_K> <part2_K> <part3_K>
    where the partN_K are portions of the same keyword, namely: part1_K goes before the atom label and usually contains the Spartan keyword, and partN_K with N>1 are appended after the atom label. For instance, this rule: 
    [#17] FFHINT= ~~10
    generated the keyword FFHINT=Cl1~~10 where "Cl1" is the atom label identified by the SMARTS [#17].
  • (optional) FREEZEATOMS the definition of atoms to freeze. SMARTS are used to identify atoms or groups to be frozen. One SMARTS query per line. Multi line block can be used.

ModifyGeometry

This task allows to move all or some specific atoms in space according to a given transformation (see also MolecularGeometryEditor). The geometrical transformation is defined as a step in Cartesian coordinates, that can be scaled one or more times. In the latter case, a sequence of new geometries is generated. The following parameters can be used to control the task:

  • INFILE pathname of the SDF file containing the molecular structure. (only SDF files with ONE molecule are acceptable!).
  • OUTFILE pathname of the SDF file where results are to be written.
  • CARTESIANMOVE (optional) pathname to a file defining the Cartesian components of the geometrical change. The file format consists of a list of translation vectors in Cartesian coordinates, where each vector is reported as a space-separted (X Y Z) tupla. The Cartesian move is orientation-dependent. Unless a reference substructure is also provided (see keyword REFERENCESUBSTRUCTURE), it is assumed that the 1-to-N vectors refer to the 1-to-N atoms in the molecular system provided as input and that the orientation of the molecule is consistent with that of the Cartesian move. When a reference substructure is provided (see parameter REFERENCESUBSTRUCTURE), the orientation of the Cartesian move is assumed to be consistent with that of the reference substructure.
  • CARTESIANSCALINGFACTORS (optional) one or more scaling factors (real numbers) to be appliead to the Cartesian move. The Cartesian move is applies on the initial structure per each given scaling factor. The default is 1.0).
  • REFERENCESUBSTRUCTUREFILE (optional) pathname to SDF file with the definition of the substructure to which the geometrical change has to be applied. The atom list is used to assign the Cartesian moves to the atoms of the molecular system given via the INFILE keyword.
  • VERBOSITY (optional) verbosity level.

RemoveDummyAtoms

This task allows to take a molecular representation (as SDF) and remove dummy atoms. Note that if none of the possible keyword is given nothing will be done (see also DummyObjectsHandler). The followgin parameters can be used to control the task:

  • INFILE pathname of the SDF file containing the molecular structure. (only SDF files with ONE molecule are acceptable!).
  • OUTFILE pathname of the SDF file where results are to be written.
  • CARTESIANMOVE (optional) pathname to a file defining the Cartesian components of the geometrical change. The file format consists on af list of vectors in Cartesian coordinates (one space-separted 3-tupla X Y Z). Unless a reference substructure is also provided (see keyword REFERENCESUBSTRUCTURE), it is assumed that the 1-to-N vectors refer to the 1-to-N atoms in the molecular system provided as input.
  • CARTESIANSCALINGFACTORS (optional) one or more scaling factors (real numbers) to be appliead to the Cartesian move. The Cartesian move is applies on the initial structure per each given scaling factor. The default is 1.0).
  • REFERENCESUBSTRUCTUREFILE (optional) pathname to SDF file with the definition of the substructure to which the geometrical change has to be applied. The atom list is used to assign the Cartesian moves to the atoms of the molecular system given via the INFILE keyword.
  • VERBOSITY (optional) verbosity level.

  • INFILE: pathname of the SDF file containing the structure.
  • (optional) LINEARITIES: with any value makes the tool remove dummy atoms used to represent linear arrangements of atoms with internal coordinates. Provide any value to this keyword to satisfy the syntax, but be aware that there is no value that dis-activate the option (e.g., LINEARITIES: NO or LINEARITIES: OFF will not dis-activate the option) It is assumed that such dummy atom are explicitly bonded to only one other atom.
  • (optional) MULTIHAPTO: with any value makes the tool remove dummy atoms used to describe multihapto systems. Provide any value to this keyword to satisfy the syntax, but be aware that there is no value that dis-activate the option (e.g., MULTIHAPTO: NO or MULTIHAPTO: OFF will not dis-activate the option) It is assumed that a dummy atom is places somewhere in the ligands (i.e., the centroid, usually) and connected to the metal and all the atoms contributing to the metal-ligand bond. For instance, one Du atom connected to the 5 carbon atoms of a cyclopentadienyl ligand and to the metal atom.
  • (optional) DUSYMBOL: the symbol to be used to identify a specific type of dummy atom to remove. Compatible with both LINEARITIES and MULTIHAPTO.
  • OUTFILE: pathname of the SDF file where the result is to be written
  • VERBOSITY: verbosity level

RicalculateConnectivity

Allows to calculate the connectivity of atoms based on the ratio between inter atomic distance and sum of van der Waals radii. That is, if two atoms are closer than the sum of their van der Waals radii minus a tolerance value (reported as percentage), then the two atoms are considered as bonded atoms. Note that the bonds existing in the input file will not be removed, and currently the new bond is always a SINGLE bond; no attempt to guess the bond order is made (see also ConnectivityGenerator). The following parameters are used when performing this task:

  • INFILE: path/name of the SDF file containing the structure (only SDF files with ONE molecule are acceptable for the moment)
  • TOLERANCE: a new bond is added only when the distance between two atoms is below the sum of their v.d.W. radii minus the tolerance (as percentage). Values between 0.0 and 1.0 should be used.
  • OUTFILE: path or name of the SDF file where the result is to be written
  • VERBOSITY: verbosity level

AddBondsForSingleElement

Attempts to add bonds around atoms of a specific element. The new bonds can be formed with atoms of any element. A bond is added if two atoms are closer than the sum of their van der Waals radii minus a tolerance value (reported as percentage). An additional tolerance can be defined for atoms sharing a common, bonded neighbor (2nd shell neighbors one to each other - 1,3 relation. Note that the bonds existing in the input file will not be removed. (see also ConnectivityGenerator)

  • INFILE: path/name of the SDF file containing the structure (only SDF files with ONE molecule are acceptable for the moment)
  • TOLERANCE: a new bond is added only when the distance between two atoms is below the sum of their v.d.W. radii minus the tolerance (as percentage). Values between 0.0 and 1.0 should be used.
  • (optional)TOLERANCE2NDSHELL: additional contribution to the tolerance of atoms in 1-3 relation (i.e., when evaluating the distance r_AC in A-B-C)
  • TARGETELEMENT: element symbol of atoms to which bonds are to be added
  • OUTFILE: path or name of the SDF file where the result is to be written
  • VERBOSITY: verbosity level

ReorderAtomList

This task is meant to change the order of atoms in an atom container. If no criterion is given than the reordered list will start from the first atom encountered in the original list for each continuously connected network of covalently bonded atom, and proceed following the connectivity. The user can request to start the reordered list from a particular type of atom (i.e., the source atom) which is identified by means of SMARTS queries. The following parameters are needed to perform this task (see also MolecularReorderer):

  • INFILE path or name of the SDF file containing the structures to reorder (only SDF files with ONE molecule are acceptable!).
  • OUTFILE path or name of the SDF file where results are to be written.
  • SOURCESMARTS (optional) list of SMARTS queries (strings, blank space separated or multi line block) defining the source atoms: the first atoms in the reordered system. Use of this option implies that for each continuously connected networks of atoms (e.g., typically a single molecule) there is at least one source atom. If no source atom is found for one or more networks the reordering cannot be performed according to the request of the user and the task is terminated with an error message. If more than one atom matches a single SMARTS query, and these candidate source atoms belong to the same network of covalently bonded atoms, the reorganization of the atoms list will start from the atom that, among the candidates, has the lowest index in the original list. If the matched atoms belong to different networks, then they are use each for reorganizing the network (i.e., molecule) they belong to. In case multiple SMARTS queries match atoms of the same molecule, the priority is given according to the order of SMARTS queries specified by the used in this option (i.e., decreasing priority is assumed).
  • VERBOSITY (optional) verbosity level.

MeasureGeomDescriptors

Measures geometrical properties such as inter atomic distances, angles, and dihedral angles. These quantities are calculated for groups of atoms identified by means of SMARTS queries, thus allowing to specify either generic or specific types of atoms (see also MolecularMeter). Parameters to be provided to perform this task are:

  • INFILE pathname of SDF/XYZ file containing the structures Multiple structures can be provided (better if mols have different name/title)
  • ATOMINDEXES single or multi line argument (see ParameterStorage) providing the definitions of the quantities to be measured. Each line codifies for a single quantity. All lines should respect the following format:

    key [AtomIndex#1] [AtomIndex#2] ([AtomIndex#3] ([AtomIndex#4]))

    where:
    • key is a string identifying the type of quantity. The string must start with DIST for inter atomic distances, ANG for angles, and DIH for dihedral angles.
    • [AtomIndex#1] ... [AtomIndex#4] are the atom indeces (1-based). Note that the number of indeces depends on the type of quantity to be measured. Moreover, note that the list is ordered.
  • SMARTS single or multi line argument (see ParameterStorage) providing the definitions of the quantities to be measured. Each line codifies for a single quantity. All lines should respect the following format:

    key [smarts1] [smarts2] ([smarts3] ([smarts4]))

    where:
    • key is a string identifying the type of quantity. The string must start with DIST for inter atomic distances, ANG for angles, and DIH for dihedral angles.
    • [smarts1] ... [smarts4] are single atom SMARTS used to identify the target atoms. Note that the number of single atom SMARTS queries depends on the type of quantity to be measured. Moreover, note that the list is ordered, meaning that, for instance, to calculate the angle between the three atoms A-B-C, the SMARTS will have to be provided with the exact order [A] [B] [C].
  • (optional)ONLYBONDED: TRUE option to restrict analysis to sequences of bonded atoms.
  • VERBOSITY verbosity level

AnalyzeVDWClashes

This task allows to analyze inter atomic distances and identify atom clashes. Being rA and rB the v.d.w radii of two atoms, and d the inter atomic distance, AtomClashAnalyzer defines an atom clash if the overlap between the two atoms is larger than a specific cutoff.

overlap = rA + rB - d - allowance

where allowance is a value in angstroms that allows to specify that some atoms can came closer to each other that the sum of their van der Waals radii.
Parameters needed by the AtomClashAnalyzer:

  • INFILE name of the SDF file containing the structures Multiple structures can be provided (better if mols have different name/title)
  • (optional) TARGETSMARTS single or multi line (see ParameterStorage) argument providing the list of SMARTS that identify the atoms to be included in the analysis.
  • (optional) CUTOFF specifies the value of the cutoff (default: 0.6)
  • (optional) ALLOWANCE specifies the value of the allowance (default: 0.4)
  • (optional) ALLOWANCE13 specifies the value of the allowance between atoms in 1-3 relationship. This value overwrites the default allowance for atoms in 1-3 relationship.
  • (optional) ALLOWANCE14 specifies the value of the allowance between atoms in 1-4 relationship. This value overwrites the default allowance for atoms in 1-4 relationship.
  • (optional) CUSTOMALLOWANCE specifies the value of the allowance between atoms defined by SMARTS strings. This option can set specific allowance values for specific pairs of atom types. Custom allowances can be defined by single or multi line argument (one custom allowance per line) according to the following format:

    [smart1] [smart2] allowance

  • VERBOSITY verbosity level

GenerateBasisSet

Used atom matching rules to build the basis set for quantum chemical calculations. This task can be embedded in other tasks, i.e., the preparation of an input for Gaussian, or run on its own. In the patter case the following parameters can be deployed:

  • ATOMSPECIFICBASISSET is used to define the atom-matching rules for assigning a specific basis set to one or more atoms in the system. One rule for each line. The syntax of each line is defined in the BSMatchingRule, and that for defining a multi line Parameter in the ParameterStorage's documentation.
  • (optional) ALLOWPARTIALMATCH allows to tollerate partial basis set that do not cover all atoms in the molecular structure. Any value other than true will forbid partial matches.
  • (optional) VERBOSITY verbosity level.
  • (optional) INFILE name of a molecular structure file containing molecule/s to be processed.
  • (optional) OUTFILE name of the output file on which the basis set will be written.
  • (optional) FORMAT is used to specify the format used when returning or writing the resulting basis set.
